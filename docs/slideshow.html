Overview

What the BAM Mutual Aid System V2 reference implementation is, and the spec it implements.

What it is

A standalone, self-hostable reference implementation of the BAM V2 spec.

A single Python service implementing the BAM Mutual Aid System V2 specification: intake, distribution outreach, check-in, no-show handling, request auto-expiration, PII scrubbing, and website metrics for a mutual-aid group.

BAM = Bushwick Ayuda Mutua, a Brooklyn neighborhood mutual-aid group that runs appointment-based goods distributions. It implements the spec in bushwickayudamutua/specs PR #1; the repo is github.com/bushwickayudamutua/bam-v2 (public).

Operational scale: 3 appointment-based distributions per week, ~60 appointments each, outreach targeting 240 people at a ~25% confirmation rate.

The problem V2 addresses

Spec section 1: technical debt, manual steps, data privacy.

The spec's problem statement: the current system has technical debt, relies on manual intervention for key workflows, and needs better data-privacy practices.

  • Manual steps — text-blast targeting requires hand-built Airtable views; post-distro inventory is informal text-based reporting.
  • Data privacy — addresses stored in plain text; PII not anonymized after fulfillment.
  • Edge cases — multiple households sharing one phone break deduplication; multiple phones per household create duplicate households.
  • Concessions — the 14-day expiration window (30 for pots & pans) may be too short for some needs.

No existing platform fit

Platform research: 62+ platforms surveyed, none met requirements.

The platform research (platform-research-summary.md, dated 2025-11-29) surveyed 62+ platforms and concluded no platform meets BAM's requirements out-of-the-box. Closest match, Crown Heights Mutual Aid App, scored ~70% but was "severely undertested" and lacked multi-language intake, appointment scheduling, and auto-expiration.

What made BAM's needs unique (per the research):

  • Household-based tracking with phone number as the unique key.
  • Multi-language intake across 11 languages.
  • Appointment-based distributions with 25%-confirmation planning.
  • Request-type-specific auto-expiration (14 vs 30 days).

Its recommendation: enhance the existing Airtable stack rather than replace it.

How this repo relates

A reference implementation of the same spec, not a replacement.

The research recommends enhancing the Airtable stack; this repo is neither that path nor a rip-and-replace. It is a reference implementation of the same V2 spec as a single Python service, so the spec's flows are concrete and testable end-to-end.

Two open questions for the BAM team

Spec ambiguities flagged in SPEC-MAPPING.md, not silently resolved.

  • Intake-row deletion. Spec 6.1 step 6 says "deletes Intake Table row", but the spec's own sequence diagram and Households schema link submissions to the household. This implementation keeps and scrubs the rows instead (SPEC-MAPPING decision 1).
  • A6 "no longer needs goods". The outreach flowchart routes it to Timeout; the 6.4 table row says "mark complete". This implementation follows the flowchart — Timeout with a note — to avoid inflating fulfillment metrics (decision 4).
Both diverge from (or resolve a contradiction in) the spec text and are flagged for the team to confirm rather than silently chosen.

Routes are thin: resolve a session, call a service function, map errors.

  • get_session() is a FastAPI dependency that yields a Session from a shared engine and closes it via with.
  • CLI commands open their own session explicitly and print a JSON report:

Engine & DB config

bam/db.py: one lazily-built engine, SQLite-aware connect args.

bam/db.pydef get_engine() -> Engine:
    global _engine
    if _engine is None:
        connect_args = {}
        if settings.database_url.startswith("sqlite"):
            connect_args["check_same_thread"] = False

  • _engine is a module-level singleton; set_engine() lets tests swap in an in-memory SQLite engine.
  • init_db() imports bam.models to register tables, then runs SQLModel.metadata.create_all(...).

Error mapping: NotFoundError -> 404

Classification by exception class, never by message text.

bam/errors.pyclass NotFoundError(ValueError):
    """An id passed to a service did not resolve to a row."""
bam/api/routes/__init__.pydef value_error_to_http(exc: ValueError) -> HTTPException:
    status_code = 404 if isinstance(exc, NotFoundError) else 400
    return HTTPException(status_code=status_code, detail=str(exc))
  • Services raise NotFoundError for unknown ids and plain ValueError for everything else; routes call value_error_to_http(exc).
Why class, not substring: messages embed user-supplied values, so matching text would let a payload like outcome="number not found" masquerade as a missing household.

Household: key fields

The record a request hangs off

  • phone_number — E.164, index=True, unique=True; the natural key for dedupe
  • phone_hash — sha256(E.164), indexed; survives scrubbing so a re-request reconnects to history
  • languages: list[str] — stored as JSON (sa_column=Column(JSON))

  • Outreach flags: last_texted, last_called, last_attended, needs_delivery, needs_email_outreach
  • Relationships to requests, social_service_requests, form_submissions

Request & SocialServiceRequest

Shared status/date shape; different extras

  • type: str — canonical key from bam.request_types, indexed; a label property resolves via label_for(self.type)
  • household_id — foreign_key="households.id"
  • Timestamps: request_opened_at, status_last_updated_at (UTC); processing_date: date | None set on Delivered/Timeout
  • Request only: delivery address fields for furniture/large items — street_address, city_state, zip_code, geocode, address
  • SocialServiceRequest only: internet_access: list[str] (JSON) and roof_accessible: bool — for the NYC Mesh install path

Processing Date: the +14 / +30 formula

apply_status_change() — models.py

Airtable's Processing Date formula is recomputed in code when status changes. The base date is the local business date (local_date(now)), so an evening distro isn't recorded under the next UTC day:

apply_status_change (spec section 4)if status == RequestStatus.DELIVERED:
    days = (
        expiry_days_for(request.type)
        if isinstance(request, Request)
        else default_expiry_days()
    )
    request.processing_date = local_date(now) + timedelta(days=days)
elif status == RequestStatus.TIMEOUT:
    request.processing_date = local_date(now) + timedelta(days=default_expiry_days())
else:
    request.processing_date = None
  • Delivered: +14 days (+30 only for Pots & Pans, via expiry_days_for)
  • Timeout: +14 days; social-service requests always use the 14-day window

Privacy provenance: hash, anonymize, airtable_id

Three columns that make scrubbing and migration reversible-enough

  • phone_hash — set at intake and preserved when the scrub nulls phone_number; a returning household reconnects by hash
  • anonymized_at: datetime | None — timestamp the household was scrubbed; a re-request clears it back to None
  • airtable_id — indexed, unique; ties each row back to its source Airtable record for idempotent import

The scrub (scrub_expired_pii) backfills the hash before nulling PII:

bam/services/privacy.pyif household.phone_number and not household.phone_hash:
    household.phone_hash = hash_phone(household.phone_number)
_null_fields(household, ("phone_number", "name", "email", "email_error", "notes"))
household.anonymized_at = now

hash_phone is sha256(e164).hexdigest() (bam/validation.py).

Airtable lookups/rollups computed in code

No stored formula columns — recomputed by query

  • Delivered Request Types (Household lookup): build_checkin_view queries distinct type where status is DELIVERED, for both request kinds, and concatenates them into delivered_request_types
  • Open Request Types (rollup): outreach.py computes sorted({r.type for r in open_requests}) per household, plus oldest_open_request_at = min(...) for outreach ordering
  • Processing Date / expiry: derived by apply_status_change, not stored as a formula
These are query-time derivations, so there is no stale cached column to reconcile — the tradeoff Airtable's rollups made.

Normalized-schema decisions

Where V2 deliberately departs from the Airtable schema (models.py docstring)

  • Fulfilled Request Count: one row per (date, request_type) with a UniqueConstraint("date", "request_type") — replaces Airtable's ~50-column-wide table
  • Zip codes stored as strings even though Airtable declares "number", so leading-zero zips survive; the intake schema coerces numeric input (_coerce_zip)
  • Legacy fields dropped: Legacy First/Last Date Submitted and Legacy Date Submitted are omitted — a V1 import maps the legacy date onto request_opened_at
  • Timestamps UTC; business dates derived via local_date in the configured timezone

Intake Flow

Assistance-request form to Household + Request rows (spec 6.1)

The pipeline: submit_form then process_submission

bam/services/intake.py, spec 6.1 steps 1-7

Two functions in bam/services/intake.py split storage from processing; intake_and_process chains them, and the API does exactly that.

  • submit_form stores the raw FormSubmission row verbatim, stamping created_at (steps 1-2).
  • process_submission validates, matches/creates the household, links the submission, opens Request and SocialServiceRequest rows, then sets processed_at (steps 3-7).
  • process_pending runs every submission with processed_at IS NULL, ordered by id.

Note: spec 6.1 step 6 says delete the intake row; per SPEC-MAPPING decision 1 the row is instead kept and linked to the household, and the privacy scrub clears its PII later.

Validation & flags: phone, email, hash

bam/validation.py

Each submission's phone and email pass through bam/validation.py before matching.

  • validate_phone parses to E.164 with default region US; returns PhoneValidation(normalized, valid, international). A number valid but outside the US sets international=True.
  • validate_email_address returns EmailValidation(normalized, error); a bad address yields error, not an exception.
  • hash_phone(e164) is sha256 hex — the stable key that lets an anonymized household reconnect on re-request.

Outcomes land on the household as spec flag fields: invalid_phone_number (not phone.valid), intl_phone_number (phone.international), and email_error.

Household matching decision tree

phone -> hash -> invalid-raw -> create

Matching order in process_submission, first hit wins:

  1. Valid phone: find by exact Household.phone_number == phone.normalized.
  2. Else phone_hash reconnect: match phone_hash == hash_phone(phone.normalized); if found, restore phone_number and clear anonymized_at (an anonymized household comes back).
  3. Invalid phone: match phone_hash == hash_phone(raw_phone) (raw string, stripped); if found, clear anonymized_at.
  4. Invalid-phone fallback: exact raw-string match against a prior submission's phone_number that already has a household_id, lowest id first (CONTRACT intake bullet 2).
  5. No match: create a new household.
Note: invalid-phone households store phone_number=None but keep phone_hash = sha256(raw string) so dedup and post-scrub reconnection still work.

Update vs create; language merge

spec 6.1 CREATE / UPDATE branches

created_household = household is None selects the branch.

Create

  • Sets name, email/email_error, languages, the phone flags, and (valid phone) phone_number.
  • phone_hash = hash of normalized phone if valid, else of the raw string.

Update (match)

  • Overwrites name only if the submission supplies one.
  • Records email_error; keeps the last good email — a re-request typo does not erase it.
  • Merges new languages onto existing (append if not already present).
  • Refreshes phone flags and, if normalized, phone_hash.

Either way the submission is linked via submission.household_id = household.id.

Requests: one Open per goods type

normalize, dedup, furniture address, bed details

Goods keys are gathered from three form fields — request_types, kitchen_items, furniture_items — each run through normalize_type (resolves keys, trilingual labels, label segments, item aliases).

  • Unresolved values, or ones that resolve to a social-service key, are pushed to unknown_types (a service in a goods field is malformed input).
  • Within one submission, duplicate goods keys are collapsed.
  • If the household already has an Open Request of that type (_has_open), the type is added to skipped_duplicate_types and no row is created.
  • New rows: status=RequestStatus.OPEN, request_opened_at=now.

Furniture-category types copy street_address, city_state, zip_code and a joined address; the furniture type appends its furniture_items to notes. Bed-detail types (mattress/bed/crib/furniture) append bed_details.

Social service requests & timestamps

internet/roof; Processing Date NULL while Open

Each value in social_service_requests is normalized; if it is not a social-service key it goes to unknown_types. Within-submission duplicates are skipped, and an existing Open row of that type routes the key to skipped_duplicate_types.

  • New SocialServiceRequest: status=OPEN, plus internet_access (list), roof_accessible (bool), and the address fields.
  • Both request tables set request_opened_at = now (Request Opened At = created time).
  • Processing Date is left NULL while a row is Open — it is stamped only on delivery/timeout, not at intake.
Note: reprocessing is guarded — if processed_at and household_id are already set, the call returns IntakeResult(already_processed=True) with no new rows.

Outreach Flow

Spec 6.2 + outreach flowchart + error paths A4-A6

build_outreach_list — the filtered candidate list

bam/services/outreach.py, spec 6.2 step 1

The admin reviews and approves the outreach list for a distro. build_outreach_list() produces the candidate set. A household qualifies only if all of these hold:

  • Has at least one Open goods request (spec: "available supplies match")
  • Has a phone number and is not flagged invalid_phone_number
  • No currently-Booked appointment (appointment_status is null or not BOOKED)
  • Language overlap with the distro's languages filter, when one is given
  • last_texted / last_attended older than the exclusion windows (spec: "not recently attended")
A request_types filter narrows both the SQL join and the per-household fulfillable check, so only households whose open requests match the distro's available supplies are returned.

The core filter, verbatim

SQL predicate + ordering

bam/services/outreach.py — statement predicateselect(Household)
    .join(Request)
    .where(
        Request.status == RequestStatus.OPEN,
        col(Household.phone_number).is_not(None),
        col(Household.invalid_phone_number).is_(False),
        or_(
            col(Household.appointment_status).is_(None),
            Household.appointment_status != AppointmentStatus.BOOKED,
        ),
    )
    .distinct()

Recency windows are applied only when > 0 (0 disables a window):

texted_cutoff = today - timedelta(days=exclude_texted_within_days)
# last_texted IS NULL OR last_texted <= texted_cutoff

TwilioSMSProvider — errors become result rows

bam/sms/twilio_provider.py

def send(self, to: str, body: str) -> SMSResult:
    try:
        message = self._client.messages.create(
            to=to, from_=self._from_number, body=body)

Partial fulfillment (A1)

out of stock? leave the request Open

There is no partial-fulfillment state (SPEC-MAPPING decision 6, spec section 7 design decision). Because fulfill_requests only touches the ids passed in, an out-of-stock item is handled by simply not including its id:

  • Fulfilled ids -> Delivered; every other open request of that household stays Open.
  • Keeping it Open preserves the household's place in the outreach queue - the spec's stated reason: "Do not mark as fulfilled to prevent deprioritization."

Auto-expiration & PII privacy

Stale requests time out on their window; inactive households are anonymized while keeping a phone hash so re-requests reconnect

Auto-expiration of stale requests

expire_stale_requests() - spec 2 / 4 / 6.1 step 7

expire_stale_requests(session, now=None) times out every Open request whose window, measured from request_opened_at, has elapsed:

bam/services/expiration.pywindow = timedelta(days=expiry_days_for(request.type))
if now - _as_utc(request.request_opened_at) > window:
    apply_status_change(request, RequestStatus.TIMEOUT, now=now)
  • Goods use the type's window: 14 days standard, 30 days for pots_pans (expiry_days_for).
  • Social-service requests always use the 14-day window (default_expiry_days()).
  • _as_utc treats naive datetimes as UTC (SQLite drops tzinfo on round-trip).

Returns an ExpirationReport of timed-out ids. Cron: daily bam expire; endpoint POST /jobs/expire.

The booked-only-if-current exemption

don't expire someone already scheduled - but don't let a stale booking be immortal

Before scanning requests, the pass collects households to spare:

bam/services/expiration.pybooked_household_ids = set(session.exec(
    select(Household.id).where(
        Household.appointment_status == AppointmentStatus.BOOKED,
        Household.appointment_date >= local_date(now),
    )
).all())
  • A currently Booked household is skipped - expiring someone already scheduled would defeat the queue (contract rule 3).
  • The appointment_date >= local_date(now) clause is deliberate (decision 8): a dangling Booked status from a distro whose no-show pass never ran must not make its requests immortal.
Why it matters: open requests block household anonymization, so an immortal request would also retain PII indefinitely.

PII scrub - three passes

scrub_expired_pii() - spec goal: hash sensitive data; background 8: V1 never anonymized after fulfillment

scrub_expired_pii(session, now=None, retention_days=None) runs three passes (retention defaults to settings.pii_retention_days, 30 days; cutoff = now - retention_days) and returns a ScrubReport with counts.

  • Pass 1 - closed requests
  • Pass 2 - anonymize inactive households
  • Pass 3 - old form submissions
  • gated on processing_date
  • gated on updated_at + no open requests
  • gated on created_at + processed

All passes are time-frozen via now and time-based, so the daily bam scrub-pii job (POST /jobs/scrub-pii) is safe to re-run.

Pass 1: scrub closed requests

Delivered/Timeout requests past their processing date lose address + notes

For requests whose status in (DELIVERED, TIMEOUT) and processing_date < today, the PII fields are nulled:

  • Goods (_REQUEST_PII_FIELDS): street_address, city_state, zip_code, geocode, address, notes.
  • Social (_SOCIAL_REQUEST_PII_FIELDS): street_address, city_state, zip_code, address, notes.
  • Closed social requests also clear internet_access (decision 13): a household's internet situation is kept only while the request is actionable.

_null_fields returns whether anything changed, so only genuinely-scrubbed rows bump updated_at and increment the report count.

Pass 2: anonymize inactive households

the privacy story - keep the phone hash so re-requests reconnect

A household is anonymized only if it has no open requests of either kind, is not already anonymized, and its updated_at is older than the cutoff:

bam/services/privacy.py, pass 2if household.phone_number and not household.phone_hash:
    household.phone_hash = hash_phone(household.phone_number)
_null_fields(household,
    ("phone_number", "name", "email", "email_error", "notes"))
household.anonymized_at = now
  • The raw phone, name, email, email_error, and notes are nulled - but phone_hash is preserved (sha256 of the E.164 / raw string).
  • On a later re-request, intake finds the household by phone_hash and restores its contact info - history reconnects instead of forking a duplicate.
  • Open requests block anonymization, which is exactly why the expiration exemption is date-bounded.

Pass 3: scrub old form submissions

kept-not-deleted intake rows lose raw PII after the retention window

For FormSubmission rows that are processed (processed_at IS NOT NULL), not yet scrubbed (scrubbed_at IS NULL), and older than the cutoff by created_at, the raw PII is nulled (_SUBMISSION_PII_FIELDS):

name, phone_number, email, notes,
street_address, city_state, zip_code
  • scrubbed_at = now marks the row done so it is not re-scanned.
  • This realizes SPEC-MAPPING decision 1: the spec's 6.1 step 6 says delete the intake row, but its own diagram links submissions to households - so BAM-v2 keeps the row and clears its PII, preserving an auditable intake history without raw data.
Open question for the BAM team: if hard deletion is truly wanted, it is a small change in bam/services/privacy.py.

The migration command

The importer migrates the real production base appjIo54Z8MWrqhlI — the base section 4 of the spec was documented from, so field names largely match verbatim.

  • Pulls a raw JSON snapshot first, then imports from it (auditable, offline re-runnable).
  • Idempotent: every row carries its airtable_id; re-runs update in place.
  • Tolerant: tables located by name, fields read through variant-aware helpers, anything unmappable is reported, never silently dropped.
bam/cli.py — import-airtable subcommand args--base-id       (default settings.airtable_base_id)

Stdlib REST client

bam/airtable.py — no third-party HTTP dependency

AirtableClient covers exactly the two endpoints the import needs — the Metadata API for table schemas and the records API with pagination — using only urllib.

  • Rate limiting: _MIN_REQUEST_INTERVAL = 0.21 keeps it under Airtable's 5 rps per base.
  • Backoff: on HTTP 429 it sleeps _RATE_LIMIT_PAUSE = 30.0 (Airtable's documented penalty) and retries, up to _MAX_RETRIES = 5.
  • Pagination: records() pages with pageSize=100, following the offset cursor until exhausted.
Needs a PAT with schema.bases:read and data.records:read scopes; a missing token raises AirtableError before any request.

Auditable snapshot, offline re-runs

dump_snapshot + SnapshotSource

The client never feeds the importer directly. dump_snapshot() writes the base's schema.json plus one <table>.json per table, and SnapshotSource reads that directory back.

  • Both AirtableClient and SnapshotSource satisfy the same RecordSource Protocol (schema() + records(table)).
  • The raw pull is the source of truth for a re-run, so a fixed import can replay against the exact bytes that were pulled.
The snapshot contains PII (names, phones, emails) — the directory is gitignored and kept local.

Tolerant field mapping

schema-driven, variant-aware, data-preserving

  • Tables are located by lowercased name against TABLE_ALIASES (e.g. social service requests or social services).
  • _first(fields, ...) accepts naming variants; _scalar() unwraps single-valued Airtable lookup lists.
  • Request types normalize to catalog keys via normalize_type; an unresolved label keeps its raw Airtable string as the type and is listed in unmatched_request_types.
  • Effective open date: request_opened_at = "Request Opened At" then "Legacy Date Submitted" then the record's createdTime.
  • Requests linking to a missing household are skipped and reported in orphaned_airtable_ids.

Idempotency & the shared-phone edge case

Every table keys on airtable_id so re-runs update rather than duplicate — Fulfilled Request Count uses its natural (date, type) key instead.

  • A household whose phone normalizes to a number an earlier record already claimed is imported without a phone (raw value kept in notes) and listed in duplicate_phone_airtable_ids — the spec's shared-phone edge case.
  • Invalid phones still hash the raw string: phone_hash = hash_phone("not a phone"), so dedup and post-scrub reconnection keep working.
  • Form submissions with a household link are marked processed (their requests already exist); unlinked ones stay pending for bam process-intake.

Consequential calls (1)

  • Missed appointments (5): missed_appointment_count increments per no-show, resets to 0 on check-in; at BAM_MAX_MISSED_APPOINTMENTS (default 2) all open goods and social requests time out.

Consequential calls (2)

decisions 8, 10, 11, 12 from SPEC-MAPPING.md

  • Date-bounded booking exemption (8): the expiration exemption applies only while appointment_date >= today, so a dangling Booked can't make requests (and PII) immortal.
  • Fulfilled counts include social services (10): deliveries of both kinds increment Fulfilled Request Count, or a delivered service would vanish from metrics.
  • Local business dates (11): timestamps are stored UTC, but count/processing/texted dates derive in BAM_LOCAL_TIMEZONE (default America/New_York).
  • Zip codes as strings (12): spec types Zip Code as a number; stored as string so leading-zero zips survive, numeric input is coerced not rejected.

Open questions for the team

Two decisions that diverge from / resolve a contradiction in the spec text

1. Intake-row deletion (spec 6.1 step 6). The spec says the system "deletes Intake Table row", but its own sequence diagram and the Households schema both link the submission to the household. We keep FormSubmission rows and scrub their PII after processing, preserving auditable history. Hard deletion would be a small change to bam/services/privacy.py — please confirm intent.
2. "No longer needs goods" (A6). The outreach flowchart routes it to marked 'timeout'; the 6.4 table row A6 says Mark complete. We follow the flowchart (Timeout with a note) so fulfillment metrics aren't inflated by goods never handed out. A distinct "withdrawn/complete" status would be an enum addition in bam/models.py — please confirm.

Cron mapping

Mirrors the spec's section 5 jobs and the bam-automation hourly/daily crons.

crontab0 * * * *  bam website-data                # hourly: UpdateWebsiteRequestData
0 6 * * *  bam expire && bam scrub-pii     # daily: auto-expiration + PII scrub
  • Hourly: bam website-data writes open request counts per type to the website JSON (spec 5).
  • Daily 06:00: bam expire times out stale open requests, then bam scrub-pii anonymizes inactive households and clears PII from old processed submissions.
  • The spec's web-triggered send_sms is POST /outreach/blast (or bam blast from a shell) — not a cron job.

01

What we're deploying

Four moving parts, one shared dataset

The system in one slide

Every device holds a full replica of two Automerge documents — the roster (who may sync) and the base (households, requests, distros, SMS outbox). Devices merge through a relay you run.

ComponentWhat it isRuns on
Volunteer consoleThe same six-view operator console as main, on a CRDT-backed BAM.api — works fully offline at a distroAny evergreen browser (phone or laptop), served as a static site
Subduction relayStore-and-forward sync server (Rust, from inkandswitch/subduction)One small VM you control
Ops runnerA Node ≥ 22.13 box enrolled like any device; runs the cron jobs + SMS gateway via src/cli.tsThe same VM is fine
Admin device(s)Bootstrap the org, enroll/revoke devices in the Roster viewAn organizer's browser

What local-first buys BAM

  • Distros work with no connectivity. Check-in, fulfillment, and intake run against the device's own replica; everything merges when the device is back online.
  • No central database to keep alive. The relay only stores and forwards; every enrolled device is a full backup.
  • Access control is per-device, not per-password. Each device has an Ed25519 key; the roster says who syncs, and revocation cuts a device off from all future syncs.
  • Same operations, same console. The views are copied verbatim from the server variant — volunteers see the identical UI.

Honest status before we deploy

  • This is an experiment branch on pre-release research software — @automerge/automerge-repo@2.6.0-subduction.40, pinned exactly.
  • Verified end-to-end (2026-07-06): device B joins by roster URL through a locally-built relay and receives the org data; 41 tests; full production snapshot imports in ~4s with counts identical to the Python importer.
  • To build during rollout: the Twilio wrapper around the outbox-drain primitive (~50 lines; the Python variant's provider is the reference).
  • Not yet true: end-to-end encryption, and server-side Keyhive enforcement of delegations (today's keyhive mode approves undelegated docs — maintainers: “we just approve all for now”).
Consequence: run this as a parallel pilot next to Airtable, not as a cutover. The plan below is scoped that way.

People & key custody

  • Two org admins minimum. Admin rights live on devices, not accounts; the roster's lockout guard stops an admin revoking itself, but a lost sole-admin device still strands the org. Two admins can always recover each other.
  • One tech volunteer owns the VM: relay service, TLS renewal, cron, backups (~1–2 h/week after setup).
  • Relay key custody: the relay's signing key file defines its identity — every client pins the relay's peer id, so back the key file up and never regenerate it casually.

03

The relay server

Build once, run as a service, pin its identity

Build and first run

# one-time build (~4 min)git clone https://github.com/inkandswitch/subduction
cd subduction && cargo build --release -p subduction_cli

# stable identity: generate the key file ONCE and keep it
head -c 32 /dev/urandom | xxd -p -c 64 > /etc/subduction/key   # 64 hex chars

./target/release/subduction_cli server \
  --socket 127.0.0.1:8944 \
  --data-dir /var/lib/subduction \
  --key-file /etc/subduction/key \
  --auth keyhive
  • On startup it prints Peer ID <64 hex> — that is the relay identity every client must trust. Record it; it goes on the volunteer join card.
Never use --ephemeral-key in production: the peer id changes on every restart and all clients stop trusting the relay.

Service + TLS

# /etc/systemd/system/subduction.service[Unit]
Description=Subduction relay
After=network.target

[Service]
ExecStart=/opt/subduction/subduction_cli server \
  --socket 127.0.0.1:8944 \
  --data-dir /var/lib/subduction \
  --key-file /etc/subduction/key \
  --auth keyhive
Restart=always
User=subduction

[Install]
WantedBy=multi-user.target
# nginx: wss://sync.bam.example → relaylocation / {
  proxy_pass http://127.0.0.1:8944;
  proxy_http_version 1.1;
  proxy_set_header Upgrade $http_upgrade;
  proxy_set_header Connection "upgrade";
  proxy_read_timeout 1h;
}
  • Same nginx serves the console's static dist/ on app.bam.example.

Why the relay must be yours (verified)

  • Client roster policies protect peer-served traffic. A relay in --auth open mode serves anyone who knows a doc URL — we demonstrated a stranger pulling a doc from an open relay despite every client denying them.
  • --auth keyhive initializes the access-control stack, but in today's build undelegated docs are still approved. Enforcement tightens when Keyhive delegation ships end-to-end (Ink & Switch's e2e demo is still private).
  • Therefore, until then: the relay is trusted infrastructure. Self-host it, restrict shell access, and never point PII at a third-party or experiment relay.
The maintainers' experiment relay (sync.subduction.inkandswitch.com) approves all relay traffic and its DNS wasn't publicly resolvable when we checked — fine for toy tests if it comes online, never for BAM data.

04

Browser devices

Hosting the console and preparing volunteer phones

Host the console as a static site

# build once per release, on the VM or in CIcd local-first
npm ci
npm run build        # → dist/ (HTML + JS + WASM, ~5 MB / ~1.7 MB gzipped)
rsync -a dist/ /var/www/app.bam.example/
  • The bundle contains code only — no data, no secrets. All data lives in each device's IndexedDB and moves only device ↔ relay.
  • HTTPS is required: the device key is a non-extractable WebCrypto key, and secure-context APIs power identity and storage.
  • The local-first GitHub workflow already runs typecheck + tests + build on every push — promote its artifact if you'd rather deploy from CI.

Device requirements & storage reality

  • Any evergreen browser works; a volunteer's own phone is the target device (same mobile-first console as main).
  • Identity: minted on first visit (WebCryptoSigner), lives in IndexedDB, never leaves the device.
  • Data: the org replica also lives in IndexedDB and survives reloads and offline days.
  • Browsers may evict storage (private windows always; iOS Safari after long disuse). That is an inconvenience, not data loss — the org data is replicated on the relay and every other device.
If a device gets wiped: it comes back with a new peer id. Procedure = revoke the old id, enroll the new one, rejoin. Two minutes, no data lost.

05

Founding setup

Day 1: org, data, ops runner

Bootstrap the org (admin device)

  1. Open the console → first-run screen → Create a new org (“BAM”). The creating device becomes the founding admin.
  2. Open the Roster view: copy the roster URL — that plus the relay endpoint is how every other device joins.
  3. Immediately enroll a second admin device (another organizer's phone) — the two-admin rule.
# alternative: bootstrap from the ops box insteadalias bam-lf="node --experimental-strip-types src/cli.ts"
bam-lf org create --name "BAM" --data-dir /var/lib/bam-lf

Enroll the ops runner and import the data

bam-lf identity --data-dir /var/lib/bam-lf        # prints the runner's peer id
# → admin enrolls it in the Roster view (role: volunteer)

bam-lf org join --roster automerge:… \
  --endpoint wss://sync.bam.example \
  --relay-peer <relay 64-hex> --data-dir /var/lib/bam-lf

bam-lf import --snapshot /path/to/airtable-snapshot --data-dir /var/lib/bam-lf
bam-lf stats --data-dir /var/lib/bam-lf
  • Expected counts (verified against the real snapshot): 1,252 households · 5,390 requests · 2,382 social-service rows (1,478 mesh) · 312 fulfilled-count rows; import takes ~4 s and is idempotent (keyed by Airtable record ids).
  • Within seconds, every enrolled device's console shows the same data.
Do not run jobs expire against freshly migrated data. Migrated requests carry their original 2024–25 open dates, so expiration would time out all of them at once (we did this to a scratch replica on purpose — recovery is one idempotent re-import). Decide open-date policy with the team first.

Day-1 acceptance test

  1. Second device joins by roster URL → sees households (this exact path is covered by the relay-gated e2e test in test/sync.test.ts).
  2. Look up a real household by phone at both devices; check one in on device A; watch it appear on device B.
  3. Turn on airplane mode on a phone → check someone in → reconnect → confirm the merge landed everywhere.
  4. Revoke a test device in the Roster view → confirm its next sync is refused.

06

Onboarding volunteers

Enroll, work, offboard — all key-based

Enrolling a volunteer (2 minutes)

  1. Volunteer opens https://app.bam.example on their phone. The device mints its key and shows the peer id with a copy button.
  2. Volunteer sends the peer id to an admin (any channel — it's a public key).
  3. Admin, in the Roster view: Enroll device → paste peer id, name (“Rosa — distro phone”), role volunteer.
  4. Volunteer taps Join with the join-card fields → the console fills with the org data and stays usable offline from then on.
Name entries after the person and device (“Rosa — personal phone”) — a person with two devices is two roster entries, revocable independently.

  • Keep the admin set small (2–3 organizer devices). Everything else is volunteer.

Offboarding & lost devices

  • Offboard: admin taps Revoke on the roster entry. Every compliant peer refuses the device's future connects/fetches/puts immediately (the policy reads the live roster doc).
  • Lost/stolen device: same action, plus honesty about what revocation can't do — it cannot recall data the device already replicated. The device held a full copy of what it had synced; treat loss as a PII incident per BAM's policy, mitigated by the nightly scrub keeping old PII out of replicas.
  • Returning volunteer: re-enroll the same peer id (re-activation is supported) or a new one if the device changed.

07

Automations on the CRDT

Cron on the ops runner; merges keep it safe

How automation works here

  • The ops runner is just another enrolled device with a replica on disk. Each job opens the store, connects to the relay, mutates its replica, lingers a few seconds so the change uploads, and exits.
  • Concurrent volunteer edits are safe by construction: CRDT merges are per-field, and the domain ops are idempotent (a fulfillment applied twice counts once; expiration re-runs are no-ops).
  • The same job commands exist in the console's Admin view — cron just makes them unattended.
# the automation surface (src/cli.ts)bam-lf jobs expire                      # daily — 14/30-day windows
bam-lf jobs scrub-pii                   # daily — PII retention
bam-lf jobs website-data --out FILE     # hourly — open counts JSON
bam-lf outbox drain [--mark-sent]       # SMS gateway primitive

Crontab for the ops runner

# /etc/cron.d/bam-lf — Node 22 on PATH; a wrapper keeps lines readable:
# /usr/local/bin/bam-lf:  #!/bin/sh
#   cd /opt/bam-v2/local-first && exec node --experimental-strip-types \
#     src/cli.ts "$@" --data-dir /var/lib/bam-lf# hourly: publish open request counts where the website reads them
0 * * * *  bam  bam-lf jobs website-data --out /var/www/request_data.json

# daily 06:00: auto-expiration, then the PII scrub
0 6 * * *  bam  bam-lf jobs expire && bam-lf jobs scrub-pii

# every 5 min: drain the SMS outbox through the gateway script
*/5 * * * *  bam  /opt/bam-v2/local-first/scripts/sms-gateway.sh
  • Identical cadence to the server variant's cron mapping (hourly UpdateWebsiteRequestData, daily expire + scrub) — the spec's section 5, relocated onto a replica.

SMS: the outbox pattern

  • An outreach blast doesn't send anything by itself — it renders the messages (randomized [REQUEST_URL], [FIRST_NAME]) and queues them in the shared smsOutbox, stamping lastTexted. Any device can queue; one gateway sends.
  • The gateway loop on the ops runner: bam-lf outbox drain lists unsent messages as JSON lines → send each via Twilio → --mark-sent stamps them so no other device re-sends.
  • To build during week 2: the ~50-line Twilio wrapper between those two steps. The server variant's bam/sms/twilio_provider.py is the reference; same env vars (TWILIO_*).
Rate limiting lives in the gateway (the spec's 30-messages-then-30s), not in the queueing — queue fast, send politely.

Website data, end to end

  • jobs website-data --out /var/www/request_data.json writes the same shape as the server variant's UpdateWebsiteRequestData: {generatedAt, counts: [{type, label, count}]} with trilingual labels.
  • Point the existing website integration at that file (or rsync it wherever the current hourly job publishes).
{

08

Coexistence with Airtable

Pilot in parallel — do not cut over

Pilot posture

  • Airtable stays the source of truth for the pilot. The CRDT org is seeded from the same snapshot pipeline (bam import-airtable on main → bam-lf import).
  • Writes made in the CRDT world (check-ins, intakes) do not flow back to Airtable — there is no reverse sync.
  • Re-importing a fresh snapshot overwrites mapped fields on rows with the same Airtable ids — statuses reset to the snapshot's values. So: freeze re-imports during the pilot window, or treat the pilot as read-mostly.
Decision for the team at pilot review: keep dual-running with periodic reseeds, build a reverse exporter, or park the branch until Keyhive enforcement ships.

Runbook & timeline

Operate it, and a four-week rollout

Backups & recovery

  • Continuous backup is structural: every enrolled device is a full replica. Additionally snapshot nightly: the relay's --data-dir, the ops runner's /var/lib/bam-lf (storage + identity.key + state.json), and /etc/subduction/key.
  • Relay dies: restore data-dir + key file on a new VM, keep the DNS name → clients reconnect untouched (same peer id).
  • Everything dies: any surviving device (a volunteer phone) still holds the org; stand up a new relay, re-point clients, and the replica re-uploads.

Monitoring (lightweight)

  • journalctl -u subduction — the relay logs every connect with a peer id; unknown ids showing up repeatedly = someone probing.
  • Cron a daily bam-lf stats diff — counts moving as expected is the cheapest health check.
  • Alert if bam-lf outbox --unsent has messages older than ~1 h (gateway stuck).
  • Pin versions; upgrade the subduction packages deliberately (pre-release — read release notes, re-run the e2e test first).
